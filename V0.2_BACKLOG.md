# Prism v0.2 backlog

Features deferred from v0.1. These are real product items, not random ideas.

## P0 — first-launch onboarding scanner

When Prism launches for the first time, scan the user's machine for existing
Claude / OpenClaw setup, show a single approval modal listing what was found,
and on consent, import everything into the Prism workspace.

**What to scan for:**

| Source | Path | Action on import |
|---|---|---|
| Claude skills | `~/.claude/skills/*/` | Symlink each skill dir into Prism's skill load path |
| Claude memory | `~/.claude/projects/*/memory/MEMORY.md` | Detect each project's MEMORY.md, offer per-project import |
| OpenClaw workspace | `~/.openclaw/workspace/*` | Symlink workspace files (SOUL.md, IDENTITY.md, USER.md, scheduled-jobs.md) |
| OpenClaw skills | `~/openclaw-workspace/skills/*` | Symlink each skill |
| Obsidian vault | `~/Documents/Obsidian Vault/` (and any other vault declared in `.obsidian/daily-notes.json`) | Symlink → Prism workspace as `obsidian-vault` |
| Scheduled tasks (launchd) | `~/Library/LaunchAgents/ai.flexhaul.*.plist`, `com.flexhaul.*.plist`, `ai.openclaw.*.plist` | Just enumerate + display; don't move |
| MCP servers | `~/.openclaw/openclaw.json` mcp.servers entries | Mirror into Prism's MCP config |

**UX:**

```
┌─────────────────────────────────────────────────────┐
│  Welcome to Prism                                   │
│  ────────────────                                   │
│                                                      │
│  We found existing AI tooling on your machine.      │
│  Import to skip the setup?                          │
│                                                      │
│  ☑ 17 Claude skills (~/.claude/skills/)             │
│  ☑ 3 Claude project memories (MEMORY.md)            │
│  ☑ 11 OpenClaw skills (~/openclaw-workspace)        │
│  ☑ Obsidian vault (~/Documents/Obsidian Vault/)     │
│  ☑ 12 scheduled jobs                                │
│                                                      │
│           [ Skip ]      [ Import selected ]         │
└─────────────────────────────────────────────────────┘
```

Implementation: new IPC handler `prism:scanExistingSetup`, returns the
list above. New IPC handler `prism:importSelected` takes the user's
selection and creates symlinks (never copies — we want changes to
propagate both ways).

## P2 — Prism-side batch orchestrator (Haiku reducer)

Today `/batch` is dispatched by the `batch-orchestrator` SKILL.md inside a
single `claude` turn — that turn's model also runs the reduce/render step.
The render step is summarization-shaped (concat + label + ≤2-sentence
synthesis); paying opus or sonnet rates for it is wasteful.

The real fix is for Prism (not the skill) to own batch orchestration:

1. Detect `/batch` (or `/parallel`, `/fanout`) in `electron/claude-client.ts::send()`
   before spawning the parent `claude` process.
2. Parse N prompts.
3. Spawn N parallel `claude --print` processes, each with its OWN
   `routeModel()` call — every prompt gets its own appropriate tier.
4. After all N return, spawn ONE final `claude --print --model haiku`
   process with the N outputs in the prompt and the rendering rules from
   `batch-orchestrator/SKILL.md`. Haiku is the right tier for the work.
5. Stream the haiku reducer's output to the renderer as the assistant
   message; emit per-subagent progress events to the existing tool-strip
   so users can see fan-out happening.

Why not now: needs careful UX (per-subagent progress, partial-failure
rendering, abort-all semantics) and a small refactor to make `send()`
spawn-multiple-aware. Skill-level fix in v0.1.21 keeps render economy
tight without the architectural change.

Win estimate: ~70-90% reduction in tokens spent on the reduce step on
batch turns (haiku 4.5 vs sonnet 4.6: ~5x cheaper input + ~5x cheaper
output; reduce-step output is the largest chunk).

## P3 — Sidebar with parallel-batch result tabs

## Deferred — Tier 2 roadmap items that don't fit current architecture

The 2026-05-08 roadmap proposed four Tier 2 items. Tier 1 (Pyodide,
gpt-tokenizer, react-diff-viewer-continued, Shiki) all shipped in v0.1.20.
Tier 2 deferred with reasons — not "later" but "here's why the current
architecture isn't the right place":

### MLX local fallback (Apple Silicon)

Real value (offline, free, fast on M-series), but the current architecture
spawns the **Claude CLI** as the inference subprocess. MLX would need:

  1. A Python sidecar process bundled into the .app (likely `mlx_lm.server`
     exposing an OpenAI-compat HTTP endpoint)
  2. Electron main-process supervision: spawn on demand, lifecycle, port
     allocation, model-cache management, error UX
  3. A model-picker rework so "Auto" / "Ollama" / "MLX" / "Claude CLI"
     route through one abstraction, not just an Electron-side conditional
  4. Quality-tradeoff UX: explicit user choice + "this is local; expect
     ~Sonnet-3.5-tier output" caveat

That's not a wire-up; that's a multi-week reshape of the inference layer.
Picking it up after streaming tokens (P1) lands so the local model has a
parallel streaming path to plug into.

### LiteLLM proxy

Makes "Auto" routing actually work, but requires either a local proxy
process (extra runtime dep, port conflict risk on user laptops) or an
OpenRouter API key (recurring user-facing cost the app currently doesn't
charge for). Either path makes Prism more complicated to install.

The "Auto" model picker is currently a stub with a single-model fallback;
revisit when a second model surface (MLX or hosted) exists, otherwise
nothing to route between.

### Promptfoo eval harness

Right tool for LLM output evaluation, wrong fit here. Prism's LLM behavior
is delegated entirely to the Claude CLI — there's no in-app prompt to
regression-test. The deterministic surface that actually drifts between
releases is `artifacts.ts::extractArtifacts`, `parseUnifiedDiff`,
`friendlyToolName`, `estimateTokens`. The right tool for those is
**vitest** (golden-input → golden-output), not Promptfoo. Add when one of
those transforms grows a regression worth catching; Promptfoo only
becomes relevant if Prism starts owning its own system prompts.

### WebContainers (StackBlitz)

Heavier than Pyodide (full Node.js + npm tarball cache in browser, ~30MB
first load) and the licensing matters: the StackBlitz SDK is free for
non-commercial / open-source distribution but commercial use needs an
agreement. Prism is MIT and may eventually be paid (Stripe wire is on the
backlog) — license decision needs to happen before adoption. Pyodide
already covers Python; React/JSX live preview is a separate ask, defer
until a user actually requests it.
